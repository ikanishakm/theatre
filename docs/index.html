Theatre.js — Newcomer Onboarding

Theatre.js is a "motion design for the web" animation library plus a visual studio editor, all built on dataverse — a custom fine-grained functional-reactive dataflow library.

monorepo · Yarn 3TypeScriptcore: Apache-2.0studio: AGPL-3.0React 19
TL;DR

Theatre.js in one read

What it is: A library for animating anything on the web (DOM, THREE.js/r3f, plain JS values) with a Google-Docs-style visual editor. You declare animatable objects in code; you keyframe them in a studio UI; production ships a tiny runtime that plays back the saved keyframes.

The 5-package mental model (bottom → top):

  1. @theatre/dataverse — the reactive engine. Atom (state) + Pointer (typed address) + prism() (cached, auto-tracking derivation) + Ticker (per-frame batching). Everything is built on this; it depends on nothing.
  2. @theatre/core — the runtime animation engine. getProject().sheet().object({props}) builds a reactive object tree; reads keyframe state, interpolates at the playhead, exposes final values via pointers/prisms. Apache-licensed, ships to production.
  3. @theatre/studio — the optional visual editor. Owns persistent state, edits values via scrub/transaction, attaches to core at runtime through a window-global handshake.
  4. Bindings — @theatre/react (hooks: useVal/usePrism), @theatre/r3f (make THREE objects editable + a snapshot editor), theatric (zero-config useControls() panel).
  5. Cloud/sync (1.0, in-flux) — @theatre/saaz (local-first optimistic sync), @theatre/sync-server (WebSocket relay), @theatre/app (Next.js accounts/auth).

Where to start: Read packages/dataverse/src/README.md (the reactivity model), then packages/core/src/sheetObjects/SheetObject.ts (getValues(), the four-layer value merge). Run yarn && yarn postinstall && yarn playground and open one of packages/playground/src/shared/*.

The one mental model to hold

The whole system is one reactive dependency graph that flows from saved keyframes down to the screen, and core never knows the studio exists.

Hold these three ideas together:

  1. Pointers address, prisms derive, Atoms hold. State lives in dataverse Atoms. A Pointer (e.g. obj.props.position.x) is just a typed, referentially-stable address {root, path} — not a value. To get a value you resolve a pointer to a prism (val() / pointerToPrism()), a lazy cached derivation that *automatically tracks* which pointers it read. When an Atom path changes, only the prisms that actually depend on it go stale, and only on the next Ticker tick do subscribers fire (batched once per frame). This is the fine-grained reactivity that lets 60fps animation touch only the handful of nodes whose values truly changed.
  1. A prop's value is a layered merge, and the playhead is just another input. SheetObject.getValues() deep-merges four layers in order of volatility: defaults → initialValue → static overrides → sequenced keyframes. The sequenced layer reads interpolationTripleAtPosition(track, sequence.positionPrism) — so the timeline playhead is simply a value in an Atom. Move the playhead (during playback or by scrubbing) and the dependency graph re-derives every animated prop reactively, with no explicit "redraw" call.
  1. Core and studio share one Atom via a runtime handshake. Core is a self-contained runtime that reads project state through a PointerProxy. When studio loads, the two find each other through window.__TheatreJS_CoreBundle / __TheatreJS_StudioBundle (they never import each other), and Project.attachToStudio() repoints that proxy at studio's own Atom. Now studio *owns* the state; an edit (scrub/transaction) mutates the studio Atom, which is the very thing core's prisms read — so the edit re-flows through the same graph to the screen. In production you drop studio, pass a baked state JSON, and the exact same prisms play it back.

Getting started

From a fresh clone to the running playground. Trust package.json over CONTRIBUTING.md for tool versions (Node ≥18, Yarn 3 — not npm).

  1. Clone and install with Yarn 3 (NOT npm)
    git clone https://github.com/theatre-js/theatre.git && cd theatre, then run yarn. The repo pins yarn@3.6.3 via .yarnrc.yml and package.json's packageManager field, with the node-modules linker. npm install will not work. Requires Node >=18 (CONTRIBUTING.md's 'node 14 / yarn 1.22' is stale — trust package.json).
  2. Run postinstall
    yarn postinstall runs husky install (git hooks) and prisma generate for the cloud packages (@theatre/app, @theatre/sync-server). For pure animation-library work you mainly need the husky hooks, but the script touches the cloud packages regardless.
  3. Start the playground (primary dev loop)
    yarn playground runs Vite (packages/playground/vite.config.ts) on http://localhost:8082. It aliases every @theatre/* import to that package's src/index.ts, so editing core/studio/r3f source hot-updates instantly with NO per-package build. Open an existing playground under packages/playground/src/shared/ (camera, three-basic, r3f-rocket, theatric, dom, ...) or create packages/playground/src/personal/<name>/{index.html,index.tsx} (personal/ is gitignored).
  4. Run unit/integration tests (Jest)
    yarn test runs Jest over packages/*/src/**/*.test.ts (transformed by jest-esbuild, @theatre/* aliased via tsconfig paths). yarn test --watch for watch mode. The dataverse integration tests (packages/dataverse/src/dataverse.test.ts) double as runnable documentation of the reactivity API.
  5. Run end-to-end tests (Playwright)
    yarn test:e2e proxies to the playground workspace and runs Playwright over *.e2e.ts files in packages/playground/src/tests. Chromium-only, 4s per-test timeout; its webServer boots the playground on port 8082 first. Visual-regression screenshots only match inside the repo's linux Docker container.
  6. Typecheck, lint, and build
    yarn typecheck (= tsc --build devEnv/typecheck-all-projects/tsconfig.all.json) checks every package. yarn lint:all runs ESLint with the custom no-relative-imports rule (--fix rewrites ../ imports to @theatre/* aliases). yarn cli build builds everything (tsc declarations, then each package's esbuild JS + api-extractor). Pre-commit hooks run lint-staged automatically.
  7. Read the foundational source in order
    Start with packages/dataverse/src/README.md (canonical hot/cold/stale/fresh diagrams). Then packages/core/src/sheetObjects/SheetObject.ts getValues() (the four-layer merge), packages/core/src/sequences/interpolationTripleAtPosition.ts (keyframe interpolation), and packages/core/src/index.ts + packages/studio/src/index.ts (the window-global handshake).

How the playground dev loop works

packages/playground/vite.config.ts is the linchpin. It:

  • sets root to src/ and appType: 'mpa';
  • globs src/{shared,personal,tests}/*/index.html with fast-glob to build the rollup input map, plus a top-level src/index.html whose index.tsx renders a PlaygroundPage listing every playground (grouped via import.meta.glob);
  • crucially sets resolve.alias from getAliasesFromTsConfigForRollup(), which turns @theatre/core → <repo>/packages/core/src/index.ts, etc. This means Vite treats the monorepo packages as source, not externals — so editing any package's src instantly reflects in the running playground with no separate build. optimizeDeps.exclude keeps core/studio out of pre-bundling in dev for the same reason.
  • injects definedGlobals (from core/devEnv) and a process.env.BACKEND_URL resolved from packages/app/.env (defaults to https://app.theatrejs.com if absent).

To hack: create packages/playground/src/personal/<name>/index.html + index.tsx (personal/ is gitignored) or use an existing one under src/shared/ (camera, dom, three-basic, r3f-rocket, theatric, notifications, etc.).

How building works

  • yarn cli build (→ tsx devEnv/cli.ts build) first runs yarn run build:ts (the aggregate tsc --build devEnv/typecheck-all-projects/tsconfig.all.json, emitting declarations) then, sequentially, yarn workspace <pkg> run build for the publishable set: @theatre/dataverse, @theatre/saaz, @theatre/core, @theatre/studio, @theatre/react, @theatre/r3f, theatric, @theatre/browser-bundles.
  • Each package's own build typically = run-s build:ts build:js build:api-json: build:ts = tsc --build (types/.d.ts + declarationMap), build:js = tsx ./devEnv/build.ts (esbuild bundling), build:api-json = api-extractor run (rolls .d.ts + emits API JSON for docs). See packages/utils/package.json for the canonical shape.
  • browser-bundles is the <script>-tag build: packages/browser-bundles/devEnv/build.ts runs esbuild twice — an IIFE dist/core-and-studio.js (src/core-and-studio.ts imports core + studio and assigns window.Theatre = core) and a minified IIFE dist/core-only.min.js (src/core-only.ts). Consumers then do Theatre.core / Theatre.studio.

How tests run

(A separate jest.compat-tests.config.js + yarn test:compat:* exercises the published-package compatibility fixtures.)
  • End-to-end (Playwright): live in packages/playground/src/tests/<name>/ as *.e2e.ts next to their playground's index.html/index.tsx. yarn test:e2e → yarn workspace playground run test → playwright test --config=devEnv/playwright.config.ts. That config's webServer boots the playground (yarn run serve:ci on port 8082, reusing an already-running instance) before the suite. yarn test:e2e:ci runs the dot-reporter chromium-only CI variant. Some .e2e.ts files also do visual-regression screenshots (expect(page).toHaveScreenshot()), which are only stable in the repo's linux Docker container (docker-compose.yml).

Typecheck & lint

  • yarn typecheck (alias of yarn build:ts) = tsc --build devEnv/typecheck-all-projects/tsconfig.all.json, a project-references graph covering every package. --watch supported.

The lowest-level functional-reactive runtime everything is built on. Atom (immutable state with per-path change tracking), Pointer (type-safe, referentially-stable {root,path} addresses via JS Proxy), prism() (lazy, cached, auto-dependency-tracking derivations with a cold/hot/stale/fresh state machine), val() (the universal read), and Ticker (per-frame batched scheduler, usually rAF-driven). No dependencies on React, the DOM, or Theatre itself. Fine-grained: only the exact derivations that depend on a changed value recompute.

End-to-end data flow

How a value travels from getProject(...).object(...) through dataverse prisms to the screen, and how studio edits flow back.

TL;DR

Theatre.js splits every concept into a thin public wrapper (Theatre*) and an internal class (Project/Sheet/SheetObject), all backed by @theatre/dataverse Atoms (state) and prisms (lazy, cached, push-based derivations). When you call getProject('x').sheet('y').object('z', {position: types.number(0)}), you build a Project -> Sheet (+SheetTemplate) -> SheetObject (+SheetObjectTemplate) tree whose config is sanitized into prop-type configs. The current value of a prop is a prism that deep-merges defaults + initial + static overrides + sequenced values, where the sequenced layer interpolates keyframes at the sequence's current playhead position. @theatre/studio attaches at runtime via a window-global CoreBundle/StudioBundle handshake, holds all project state in its own Atom (via the Saaz store), and edits values through scrub/transaction APIs that mutate that Atom; because core reads project state through a PointerProxy pointing at the studio Atom, the change re-flows through the same prisms and re-renders reactively.

  1. DEFINE: You call getProject('x').sheet('y').object('z', {position: types.number(0)}). This builds the internal Project (+ historic/ephemeral state Atoms wrapped in PointerProxies), SheetTemplate, Sheet, SheetObjectTemplate (config sanitized via compound() into PropTypeConfigs), and SheetObject. The returned obj.props is a Pointer rooted at the SheetObject — an address, not a value.
  2. READ: val(obj.props.position) resolves the pointer through SheetObject.pointerToPrism() (SheetObject is a PointerToPrismProvider). That prism reads SheetObject.getValues(), a cached prism that deep-merges four layers in order of volatility: defaults → initialValue → static overrides (from project historic state) → sequenced values. For position with no keyframes/overrides yet, the result is the default 0.
  3. SUBSCRIBE: A consumer attaches reactivity — either onChange(obj.props.position, cb) in user code (fires on the core Ticker) or useVal(obj.props.position) in React/studio UI (subscribes via the prism's onStale and re-renders on a batched microtask, only when the value actually changed).
  4. ATTACH STUDIO: On import, core publishes window.__TheatreJS_CoreBundle and studio publishes window.__TheatreJS_StudioBundle; whichever loads second calls registerCoreBundle, exchanging CoreBits (projectsP pointer, coreExports, privateAPI). Project.attachToStudio() then repoints the project's PointerProxies at studio.atomP.historic.coreByProject[projectId], so core's getStaticValues()/sequenced-layer prisms now read the STUDIO-owned Atom.
  5. SCRUB: In the studio, dragging a prop editor calls getStudio().scrub() then scrub.capture(api => api.set(pointerToProp, newValue)). transactionApi.set resolves the SheetObject + prop config and checks getMapOfValidSequenceTracks_forStudio(): if a track exists it writes/updates a KEYFRAME (setKeyframeAtPosition), otherwise a STATIC OVERRIDE (setValueOfPrimitiveProp). These stateEditors mutate the studio's saaz draft; on commit, saaz.subscribe fires and StudioStore._atom.set(state.cell) updates the studio Atom.
  6. RE-FLOW: Because core reads project state through the PointerProxy now pointing at the studio Atom, mutating that Atom stales exactly the path's Scope listeners → the getValues() merge prism goes stale → the position prop prism goes stale. No explicit invalidation is wired.
  7. RENDER: On the next Ticker tick (for onChange) or batched microtask (for @theatre/react's order-sorted queue, wrapped in unstable_batchedUpdates), the stale prisms are freshened, getValue() returns the new value, and the listener/component is notified ONLY if the value actually changed — updating the value on screen.
  8. PLAYBACK (alternative driver): sheet.sequence.play() runs DefaultPlaybackController, which advances position = elapsedRealTime*rate into an Atom each tick. Sequence.positionPrism reads that Atom; interpolationTripleAtPosition(track, positionPrism) re-selects the bracketing keyframes and solves the bezier {left,right,progression}; getSequencedValues() interpolates and the same merge→prop→listener chain re-derives every animated prop per frame.
  9. PRODUCTION: Drop @theatre/studio, pass the studio-exported state JSON to getProject(id,{state}). Core loads it straight into the historic Atom (no proxy swap), and the identical prism graph plays back the animation — only the sequenced layer changes per frame, kept cheap by WeakMap-cached merges.
Deep dive — trace a value from object() to the screen and back

1. Construction: getProject('x').sheet('y').object('z', {position: types.number(0)})

getProject('x') (coreExports.ts): looks up projectsSingleton; if absent constructs new TheatreProject(id, config). TheatreProject's constructor (projects/TheatreProject.ts) calls setPrivateAPI(this, new Project(...)). The Project (projects/Project.ts) creates two local Atoms (onDiskStateAtom for historic state with sheetsById, and an ephemeral atom), wraps them in PointerProxy instances, and exposes this.pointers.historic / this.pointers.ephemeral from those proxies. It registers itself in projectsSingleton (this is what studio will later see). Returns this.publicApi.

.sheet('y') (projects/TheatreProject.sheet -> Project.getOrCreateSheet): creates/caches a SheetTemplate per sheetId in an Atom, then template.getInstance(instanceId) builds a Sheet (sheets/Sheet.ts). The Sheet holds an Atom<SheetObjectMap> of its objects and a TheatreSheet public wrapper. Returns sheet.publicApi.

.object('z', {position: types.number(0)}) (sheets/TheatreSheet.object): first the config object is sanitized via compound(config) (propTypes/index.ts) into a PropTypeConfig_Compound. types.number(0) (propTypes/index.ts) is itself a PropTypeConfig_Number carrying {type:'number', default:0, deserializeAndSanitize, interpolate, nudgeFn,...}. Then Sheet.createObject() calls SheetTemplate.getObjectTemplate() (creating a SheetObjectTemplate, which stores the config in an Atom and precomputes pointers like pointerToStaticOverrides into the project's historic state), and template.createInstance() returns a SheetObject (sheetObjects/SheetObject.ts) wrapped by TheatreSheetObject. The object is stored in the sheet's objects Atom. obj.props is a pointer (SheetObject.propsP) whose root is the SheetObject itself.

So the created internal objects are: Project (+2 PointerProxies, +state Atoms), SheetTemplate, Sheet, SheetObjectTemplate (+config Atom), SheetObject (+initialValue Atom +SimpleCache), each with a Theatre* public wrapper.

2. Reading the current value of position

val(obj.props.position) (coreExports.val) calls pointerToPrism(pointer). The pointer's root is the SheetObject, which is a PointerToPrismProvider, so SheetObject.pointerToPrism(pointer) runs. It returns a prism that reads val(this.getValues()) then drills to path (['position']) via pointerDeep.

SheetObject.getValues() (cached prism) computes the final value as a deep-merge of layers (see getValues() doc comment):

  1. val(template.getDefaultValues()) — a prism reading the config Atom and extracting defaults (so position = 0).
  2. val(this._initialValue.pointer) — usually empty.
  3. val(template.getStaticValues()) — reads val(this.pointerToStaticOverrides) i.e. project.pointers.historic...staticOverrides.byObject[objectKey], deserialized via the config. This is where a studio static edit lands.
  4. The sequenced layer from getSequencedValues().

Each merge uses a deepMergeWithCache with a prism.memo'd WeakMap. The merged result is wrapped in an Atom via valToAtom and its pointer returned, so callers get a pointer they can drill into.

Sequenced layer (getSequencedValues): for each valid track (template.getArrayOfValidSequenceTracks()), it builds this._trackIdToPrism(trackId). That combines the track-data pointer (project.pointers.historic...sequence.tracksByObject[objectKey].trackData[trackId]) with this.sheet.getSequence().positionPrism through interpolationTripleAtPosition (sequences/interpolationTripleAtPosition.ts). The triple {left,right,progression} is deserialized and fed to propConfig.interpolate(left, right, progression); the result is written into a values Atom at pathToProp inside a prism.effect. Because the sequenced layer is merged last, a keyframed track overrides the static/default value.

During playback: the playhead position lives in DefaultPlaybackController._state (an Atom<{position, playing}>). Sequence._positionD/positionPrism reads statePointer.position. On each frame the play loop calls gotoPosition -> _state.setByPointer(p=>p.position, t). That stales positionPrism -> stales the interpolation prism -> interpolationTripleAtPosition re-selects/recomputes the bezier triple -> getSequencedValues re-interpolates -> getValues re-merges -> the prop prism produces the new value. interpolationTripleAtPosition caches its validFrom/validTo window so it only re-selects keyframes when the playhead crosses a keyframe boundary.

Push mechanics (dataverse): a prism (prism/prism.ts) auto-collects the pointers/prisms it reads as dependencies; when an underlying Atom path changes it notifies its Scope listeners (Atom.ts), the prism's HotHandle marks itself stale and propagates staleness to dependents. onChange(pointer, cb) (used by coreExports.onChange) is prism.onChange(ticker, cb): on stale it schedules refresh via ticker.onThisOrNextTick, recomputes, and calls the listener only if the value actually changed.

  • React UI (studio panels, and user apps) using usePrismInstance/useVal from @theatre/react (packages/react/src/index.ts) get an onStale callback, which queues the component (ordered parent-before-child) and re-renders on a microtask via unstable_batchedUpdates, re-reading the fresh value.

@theatre/dataverse

The animation-optimized functional-reactive dataflow library that everything in Theatre.js is built on.

TL;DR

Dataverse is a tiny FRP (functional reactive programming) library inspired by libraries like MobX, but tuned for interactivity and animation. You put state in an Atom, refer to nested paths with type-safe Pointers, derive computed values with prism() (which auto-tracks its dependencies), read values with val(), and schedule/batch reactive callbacks through a Ticker (usually driven by requestAnimationFrame). It enables fine-grained reactivity: only the exact derivations that depend on a changed value recompute, and only the exact subscribers fire.

Where it fits

This is the lowest-level reactive runtime in the Theatre.js monorepo. @theatre/core, @theatre/studio, @theatre/react, @theatre/r3f, and @theatre/theatric all depend on it. It has essentially no runtime dependencies except a few lodash-es helpers (get, isPlainObject, last). The React bindings in @theatre/react (useVal, useAtom, usePrism) are thin wrappers over val(), Atom, and prism() plus a Ticker. Nothing inside dataverse depends on React, the DOM, or Theatre itself — it's a standalone primitive.

Core concepts

Atom
A mutable holder of an immutable state value. You replace state wholesale (set/reduce) or by path (setByPointer/reduceByPointer, which use updateDeep for structurally-shared immutable updates). Internally an Atom keeps a lazily-grown tree of Scope nodes — one per subscribed path. On every set, _checkUpdates walks only the scopes that have listeners, comparing old vs new value by reference (===) at each node, and fires identityChangeListeners only where identity actually changed. This is how fine-grained, per-path change detection stays cheap: unchanged subtrees share references so the walk short-circuits.
Pointer
A type-safe, referentially-stable reference to a path inside an atom (e.g. atom.pointer.light.position.x). It carries no value — just {root, path} metadata. Pointer<O> is a recursive mapped type so the compiler knows the type at every path. At runtime a pointer is a JS Proxy: its meta lives in a WeakMap keyed by an empty 'pointerKey' object, and every property access (.x) is intercepted to return a *cached* sub-pointer (stored in another WeakMap), which is why p.a.b === p.a.b. That referential stability is what lets React memoize on a pointer and lets prisms be cached per-pointer.
Prism
A cached, lazily-evaluated derivation built from a pure-ish function. prism(() => val(a) + val(b)) automatically discovers that it depends on a and b because reading them (via the discovery mechanism's collector stack) registers them as dependencies. Prisms compose (a prism can read other prisms), and re-run only when a real dependency changes. They're the composable, cached, Ticker-coordinatable alternative to raw onChange listeners.
Hot / Cold
A prism is COLD when nothing subscribes to it (no dependents, or only cold dependents) — calling getValue() on a cold prism recomputes from scratch each time (via calculateColdPrism, using a no-op ColdScope where effects don't run). It becomes HOT the moment it gets a dependent (a subscriber via onChange/onStale, or a hot prism that depends on it). Going hot creates a HotHandle that maintains the dependency graph and caches the value. When the last dependent leaves, the prism goes cold again and its HotHandle (and effects) are destroyed. Hotness propagates transitively up the dependency graph.
Stale / Fresh
A *hot* prism is either FRESH (its cached value is up to date) or STALE (a dependency changed since the last read, so the cache is invalid). When a dependency goes stale it calls _reactToDependencyGoingStale, which marks this prism stale and propagates 'stale' to its dependents in topological order (without recomputing — that's lazy). Reading getValue() on a stale prism triggers _recalculate, which freshens dependencies first and even skips recomputation if the possibly-stale deps turn out to have unchanged values (a cheap equality short-circuit).
Prism hooks (memo/effect/state/ref/source/sub/scope)
Inspired by React hooks but KEYED by a string instead of call-order, so they can be called conditionally or in any order. memo(key,fn,deps) caches; effect(key,fn,deps) runs side-effects with cleanup (only in hot prisms); state(key,init) returns [value,setValue] where setValue forces the prism stale; ref(key,init) is a mutable box; source(subscribe,getValue) lets a prism react to an EXTERNAL source (this is how an Atom feeds a prism); sub(key,fn,deps) creates a nested memoized prism for fine-grained sub-recomputation; scope(key,fn) namespaces hook keys.
val()
The single read primitive that normalizes pointers, prisms, and plain values into a value. Crucially, when called on a pointer/prism *inside a running prism*, the read is tracked as a dependency (because getValue() calls reportResolutionStart). Outside a prism it's just a one-off non-reactive read.
Ticker
A scheduler that decouples 'something changed' from 'run the subscriber'. Subscribers don't fire synchronously on change; instead the prism's onChange schedules a refresh onto the ticker (onThisOrNextTick). Calling ticker.tick(t) drains all scheduled callbacks once per frame. Because callbacks live in a Set, repeated changes within a frame collapse into one call (batching), and a value is recomputed at most once per tick. Typically driven by requestAnimationFrame to sync with the display refresh rate; goes dormant after ~3s of empty ticks.

Typical usage

import {Atom, prism, val, Ticker} from '@theatre/dataverse'

// 1. State in an Atom
const atom = new Atom({a: 1, b: 2})

// 2. Type-safe, referentially-stable pointers
atom.setByPointer(atom.pointer.a, 5)
atom.pointer.a === atom.pointer.a // true

// 3. A derived, auto-tracked, cached prism
const sum = prism(() => val(atom.pointer.a) + val(atom.pointer.b))

// 4. Non-reactive read
val(sum) // 7

// 5. Reactive read, batched via a Ticker
const ticker = new Ticker()
const raf = (t: number) => {
  ticker.tick(t)
  requestAnimationFrame(raf)
}
requestAnimationFrame(raf)

const unsub = sum.onChange(ticker, (v) => console.log('sum =', v))

atom.setByPointer(atom.pointer.a, 10)
atom.setByPointer(atom.pointer.b, 20)
// On the next tick, logs once: "sum = 30" (batched, computed once)

unsub() // sum goes cold again

In React (via @theatre/react):

import {useVal} from '@theatre/react'
function Light({intensityP}) {
  const intensity = useVal(intensityP) // re-renders only when this path changes
  return <div style={{opacity: intensity}} />
}

Deep dive — how it works internally

# Data flow, end to end

State lives in Atoms, addressed by Pointers

You create new Atom(initialState). The atom keeps _currentState (treated as immutable) and a _rootScope. A Scope is a node in a lazily-built tree that mirrors the paths people have subscribed to; each scope holds identityChangeListeners and children. Scopes are created on demand by _getOrCreateScopeForPath and garbage-collected (_checkForGC) when they have no listeners and no children.

To address a sub-path you use a Pointer: atom.pointer.light.intensity. A pointer is a Proxy over an empty key object; the key maps (via pointerMetaWeakMap) to {root: atom, path: ['light','intensity']}. Property access on the proxy (.intensity) is intercepted by proxyHandler.get, which builds-or-returns a cached child pointer (cachedSubPathPointersWeakMap). That cache is why pointers are referentially stable — the same path always yields the same pointer object — and the proxy + recursive Pointer<O> mapped type is why they're type-safe.

Writes: immutable + identity-based change detection

atom.setByPointer(p, v) → reduceByPointer → updateDeep(state, path, () => v). updateDeep rebuilds only the spine along path, structurally sharing every untouched subtree (returning the *same reference* when nothing under a node changed). Then set(newState) calls _checkUpdates(rootScope, oldState, newState), which recurses only into scopes that have listeners. At each node it compares oldState === newState; if equal it bails immediately (this is the payoff of structural sharing), otherwise it fires that scope's listeners and recurses into children. So a change deep in the tree notifies exactly the subscribers on that path and its ancestors, and nobody else.

Atom → Prism bridge via prism.source()

atom.pointerToPrism(pointer) returns prism(() => prism.source(subscribe, getValue)), where subscribe is onChangeByPointer(pointer, listener) and getValue reads the current value at that path. pointerToPrism() (the standalone fn) caches one prism per pointer-meta in a WeakMap, so val(somePointer) repeatedly reuses the same prism. val() on a pointer is literally pointerToPrism(pointer).getValue().

Prisms: the dependency graph

A PrismInstance starts cold: _state = {hot:false}. getValue() on a cold prism just runs the function fresh in a throwaway ColdScope (effects warn/no-op, memo just calls through). The first dependent (_addDependent, triggered by onChange/onStale, or by another hot prism reading it) flips it hot, creating a HotHandle.

The HotHandle's constructor runs the function once (with dependency tracking suppressed during setup, then getValue()). Recalculation (_recalculate) works like this:

  1. If some deps are 'possibly stale', re-read them; if none actually changed value, return the cached _lastValue (cheap short-circuit).
  2. Otherwise push a collector onto the global discovery stack, push this prism's HotScope onto the hook-scope stack, and run _fn().
  3. While _fn runs, every nested getValue() (from val(...)) calls reportResolutionStart, which notifies the collector → that dependency is added to newDeps and _addDependency wires up a 'go stale' callback.
  4. After the run, dependencies that weren't touched this time are removed, the new dependency set is stored, and each dep's current value is cached for the next short-circuit check.

When a dependency changes, it calls each dependent's _reactToDependencyGoingStale, which records the dep as possibly-stale and calls _markAsStale. _markAsStale sets _isFresh=false and propagates staleness to *its* dependents — but does NOT recompute. Recomputation is lazy: it only happens when someone calls getValue(). This is the stale→fresh transition, and it freshens in reverse-topological order (a reads its deps, which read theirs).

Subscriptions are Ticker-batched

prism.onChange(ticker, listener) adds an internal dependent that, whenever the prism goes fresh→stale, calls ticker.onThisOrNextTick(refresh). It does NOT call the listener synchronously. On the next ticker.tick(), refresh runs, reads getValue() (freshening the graph), and calls the listener only if the value actually changed (cached lastValue comparison, with a sentinel emptyObject so the first run always fires). onStale(cb) is the lower-level primitive that fires synchronously on every fresh→stale transition (used by keepHot, and by iterateAndCountTicks).

The Ticker

tick(t) moves _scheduledForNextTick into _scheduledForThisOrNextTick, then _tick swaps the set out, runs every callback, and if callbacks scheduled more 'this-or-next-tick' work it recurses (up to a hard limit of 100, warning at 10). Because callbacks are stored in Sets keyed by function identity, scheduling the same refresh twice in a frame collapses to one call — that's the batching guarantee. After EMPTY_TICKS_BEFORE_GOING_DORMANT (180) ticks with nothing scheduled, it goes dormant and stops doing work until something schedules again.

Why this design (fine-grained reactivity)

The whole point is to recompute and re-render the absolute minimum. Pointers give cheap, stable, type-safe addresses; identity-based diffing + structural sharing means a set only notifies the exact subscribed paths; prisms cache and only invalidate along real dependency edges; the cold/hot machinery means unused derivations cost nothing and clean up their effects automatically; and the Ticker batches everything to one synchronized pass per animation frame. In Theatre.js this lets a 60fps animation update only the handful of DOM/React nodes whose underlying values actually changed this frame.

Gotchas & footguns (10)
  • val()/getValue() only tracks a dependency when called INSIDE a running prism. Calling prism.getValue() (or val(pointer)) in a React render body or a plain callback is a common mistake — it reads a value but won't make anything reactive. Use useVal/usePrism in React. There's a long TODO in prism.ts about warning on this.
  • Prism hooks are KEYED by string, not by call order (unlike React hooks). Two hooks with the same key in the same prism scope collide — use prism.scope('namespace', ...) to disambiguate, or unique keys.
  • Effects (prism.effect, and therefore prism.source/prism.state machinery) only run when the prism is HOT. On a cold prism, ColdScope makes effect a no-op (it even console.warns), memo just calls the fn every time, and state's setter does nothing. So a prism you only ever getValue() without subscribing behaves differently than a subscribed one.
  • Subscribers fire on a Ticker tick, NOT synchronously. After atom.set(...), onChange listeners haven't run yet — you must ticker.tick(). In tests this trips people up constantly (see Ticker.test.ts / dataverse.test.ts). onStale IS synchronous, but onChange is deferred and batched.
  • onChangeByPointer/onChange on an Atom are plain event listeners — the docstring explicitly warns they don't compose, aren't cached, and aren't Ticker-coordinated. Prefer a prism (which uses them internally) unless you truly want a raw listener.
  • Change detection is by reference identity (===), not deep equality. updateDeep preserves references for unchanged subtrees, so a no-op setByPointer to the same value won't notify. But replacing state with a deeply-equal-but-new object WILL notify everyone, because identity changed.
  • Referring to a non-existent property via a pointer is a TypeScript error but works fine at runtime (and will create that property on set) — pointers are structurally permissive at runtime.
  • The dependency-discovery mechanism is a SINGLE shared stack stashed on window/global (__dataverse_discoveryMechanism_sharedStack). This is deliberate so that two copies of dataverse loaded in the same realm still track each other's dependencies — but it means duplicate/mismatched versions in one page can interact in subtle ways.
  • Errors thrown inside a prism function or an effect are swallowed and re-thrown asynchronously via setTimeout (safelyRun/PrismReportThrow) so they don't corrupt the prism's internal state — meaning a prism error won't crash the current call stack but will surface later as an unhandled rejection-style throw.
  • The Ticker's tick() must be called bound to the ticker instance (it checks this instanceof Ticker in dev). Pass ticker.tick carefully, e.g. requestAnimationFrame((t) => ticker.tick(t)) or bind it.

@theatre/core

The Apache-licensed runtime animation engine of Theatre.js: turns saved keyframe state into live, interpolated prop values you read via pointers.

TL;DR

@theatre/core is the small, production-shippable runtime that plays back animations authored in the studio. You call getProject(id, {state}).sheet(name).object(key, props) to get a typed object whose .props is a reactive pointer tree; under the hood core reads keyframe state, interpolates it at the sequence's current playhead position, and exposes the final value through dataverse prisms/pointers. It can run standalone (state baked in at build time) or, when @theatre/studio is loaded on the same page, hand over control of its state via a global-variable handshake so you can edit live.

Where it fits

This is the foundational runtime layer everything else sits on. It depends only on @theatre/dataverse (the reactive Atom/Pointer/prism system) and @theatre/utils. @theatre/studio is an optional peer that attaches to core at runtime via window.__TheatreJS_CoreBundle and takes over state management/persistence; @theatre/r3f, @theatre/react, and @theatre/theatric are higher-level wrappers built on core's public API (getProject, onChange, val, types). In production bundles you typically ship core alone with a pre-exported state JSON; in development you also load studio. Core never imports studio directly — it only knows it through a $____FixmeStudio placeholder type and the global handshake.

Core concepts

The object model: Project -> Sheet -> Object(props) -> Sequence
getProject(id, {state}) returns a Project (deduped per-id). project.sheet(name, instanceId='default') returns a Sheet. sheet.object(key, propsConfig) returns a SheetObject whose .props is a typed pointer tree matching propsConfig. Each sheet has exactly one Sequence (sheet.sequence) that holds the timeline/playhead for animating that sheet's objects. Sheets can be instantiated multiple times (different instanceId) sharing one SheetTemplate; likewise objects share a SheetObjectTemplate across instances.
Public facade vs. private internal (the privateAPI WeakMap)
Every user-visible class is a thin facade: TheatreProject/TheatreSheet/TheatreSheetObject/TheatreSequence. Each wraps a real internal class (Project/Sheet/SheetObject/Sequence) and looks it up via the WeakMap in privateAPIs.ts. This keeps the public surface stable/typed while the internals can change. When studio needs internals it calls privateAPI(publicThing) — that resolver is one of the CoreBits it receives.
Pointers and prisms (dataverse) map props to values
object.props is a Pointer — a typeless, comparable 'address' into a value tree (root=SheetObject, path=[...]). Reading it (via val() or onChange()) calls SheetObject.pointerToPrism(), which produces a dataverse prism (a lazily-computed, cached, subscribable derivation). val(pointer) snapshots the current value; onChange(pointer, cb, rafDriver) subscribes and recomputes on a ticker. Pointers are the public-facing handles; prisms are the reactive machinery behind them.
The four-layer value merge
A SheetObject's final value is computed in SheetObject.getValues() by deep-merging, in increasing order of volatility: (1) defaults from the prop config, (2) initialValue set via obj.initialValue= (rare, mainly r3f), (3) static overrides stored in project state (staticOverrides.byObject), (4) sequenced values (interpolated keyframes at the current playhead). Layers are ordered cheapest-to-most-volatile and each merge step uses a cached WeakMap so per-frame recomputes are fast — only the sequenced layer typically changes during playback.
Sequences, tracks, and keyframe interpolation
State stores, per object, trackIdByPropPath (which prop path -> which track) and trackData (track id -> BasicKeyframedTrack of keyframes). At read time, interpolationTripleAtPosition(trackPointer, positionPrism) finds the keyframes bracketing the current position and returns {left, right, progression}; SheetObject.getSequencedValues() then runs propConfig.deserializeAndSanitize on left/right and propConfig.interpolate(left, right, progression). Easing is a UnitBezier solved from the keyframes' handles (or a hold/step curve when type='hold'); 'connectedRight:false' means the prop stays at left with no interpolation to the next keyframe.
Playback controllers and the position prism
Sequence delegates time to a swappable IPlaybackController held in an Atom. DefaultPlaybackController stores {position, playing} in an Atom and, on play(), schedules ticks on a dataverse Ticker, advancing position by elapsed-real-time*rate within [range], honoring iterationCount + direction. Sequence.positionPrism exposes that position reactively; because interpolation reads positionPrism, moving the playhead automatically re-derives every sequenced prop. attachAudio() swaps in an AudioPlaybackController so time is driven by the Web Audio clock instead.
rafDrivers and tickers
All recomputation is driven by dataverse Tickers, fed by rafDrivers. coreTicker.ts lazily creates a default rafDriver backed by requestAnimationFrame (or a one-shot setTimeout in SSR). createRafDriver() lets you supply your own tick source (to sync with r3f/gsap/lenis, an XR frame loop, or manual stepping for rendering/tests). onChange() and sequence.play() both accept a rafDriver to control at what rate listeners/playhead update.
PropTypeConfig: shorthand to longhand
sheet.object('key', {x: 0}) accepts shorthand: a raw number/string/boolean/plain-object. compound() (via sanitizeCompoundProps -> toLonghandProp in propTypes/internals.ts) converts these into full PropTypeConfig objects (e.g. 0 -> t.number(0)). Every config carries: type, default, a typescript-only valueType, deserializeAndSanitize() (validates/coerces stored JSON, returns undefined if invalid so the default is used), and for simple types interpolate(). Composite types (compound, enum) are not directly sequenceable — only their simple leaf descendants are.
State loading and the studio handoff
In core-only mode, the {state} JSON you pass to getProject is dropped straight into the project's historic Atom and the project becomes ready on the next tick. If studio is present, attachToStudio() runs initialiseProjectState(): it compares your on-disk state against any browser-persisted (studio) state via revisionHistory, decides which wins, writes it into studio's atoms, then re-points the project's PointerProxies at studio's atoms so studio becomes the source of truth (and can persist edits). validateProjectIdOrThrow / shallowValidateOnDiskState guard the inputs.
Core<->Studio global registration handshake
index.ts calls registerCoreBundle() at import: it constructs a CoreBundle and stores it on window.__TheatreJS_CoreBundle (throwing a helpful error if a second core is detected — usually a bundler dedupe misconfig). If a studio bundle is already on window.__TheatreJS_StudioBundle, core calls studioBundle.registerCoreBundle(coreBundle). Studio later calls coreBundle.getBitsForStudio() to receive CoreBits and attach. This indirection is precisely why core stays Apache-licensed and studio-free in production bundles.

Deep dive — how it works internally

Lifecycle and data flow

1. Import side-effects. Importing @theatre/core runs index.ts, which immediately calls registerCoreBundle(). This publishes a CoreBundle instance to window.__TheatreJS_CoreBundle (in browsers; gated by global.__THEATREJS__FORCE_CONNECT_CORE_AND_STUDIO in node). The bundle carries version (from env.THEATRE_VERSION) so a duplicate-bundle situation throws a clear diagnostic. If a studio bundle is already registered on window.__TheatreJS_StudioBundle, core registers itself with it right away. This is the entire mechanism by which core and studio find each other — they never import each other's code; core only references studio through the $____FixmeStudio placeholder type.

2. Getting a project. getProject(id, config) (in coreExports.ts) checks projectsSingleton for an existing project (and in dev throws if you re-call with a different config). On miss it validates the id and, if config.state is present, validates its definitionVersion against globals.currentProjectStateDefinitionVersion ('0.4.0'), then constructs a TheatreProject, whose constructor builds the internal Project. The Project seeds two dataverse Atoms: a historic atom (the keyframe/override data — either config.state or an empty {sheetsById:{}, definitionVersion, revisionHistory:[]}) and an ephemeral atom (loading state, never persisted). Both are wrapped in PointerProxys so the underlying atom can later be swapped out for studio's.

3. Readiness and the studio race. A Project exposes ready (a promise). In core-only mode with config.state, a setTimeout(…, 0) gives studio one tick to attach; if it doesn't, the project resolves ready immediately. If studio *does* attach, attachToStudio() waits for studio.initialized, runs initialiseProjectState() to reconcile on-disk vs. browser-persisted state (using revisionHistory to detect divergence), then re-points the project's PointerProxys at studio.atomP.historic.coreByProject[projectId] and the studio ephemeral atom — making studio the live source of truth. Asset storage is initialized last. Without state and without studio, core warns/errors (SSR logs; browser throws after 1s) because nothing would animate.

4. Sheets and objects. project.sheet(name, instanceId) sanitizes the name and calls Project.getOrCreateSheet, which lazily creates a SheetTemplate per sheetId and returns a Sheet instance per instanceId. sheet.object(key, props, opts) sanitizes the path, runs compound(props) to turn the shorthand config into a full PropTypeConfig_Compound, and (on first call) creates a SheetObjectTemplate + SheetObject. Re-calling with a different config throws unless {reconfigure:true} is passed. The returned facade's .props is a Pointer rooted at the internal SheetObject with an empty path.

5. Reading a value (the merge). When you val(obj.props.position.x) or onChange(...), dataverse resolves the pointer through SheetObject.pointerToPrism(), which reads getValues(). getValues() is a memoized prism that computes the final value as a 4-layer deep-merge: defaults (template.getDefaultValues(), derived from the prop config) → initialValue (obj.initialValue=, used mainly by r3f) → static overrides (template.getStaticValues(), read from state.…staticOverrides.byObject[objectKey] and deserializeAndSanitized) → sequenced (getSequencedValues()). Each merge uses a stable cached WeakMap (via prism.memo) so most layers are reused frame-to-frame; only the sequenced layer typically changes during playback, keeping per-frame cost low. The result is wrapped in an Atom and a pointer to it is returned.

6. Sequenced values and interpolation. getSequencedValues() asks the template for getArrayOfValidSequenceTracks() — the tracks whose prop path still corresponds to a sequenceable (non-composite) prop in the current config, sorted to config order. For each track it builds _trackIdToPrism(trackId), which is interpolationTripleAtPosition(trackPointer, sequence.positionPrism). That function locates the keyframes surrounding the current playhead and returns {left, right, progression}: before the first keyframe → that keyframe's value at progression 0; after the last → likewise; between two connected keyframes → a UnitBezier solved from the keyframes' bezier handles (or a Math.floor step for type:'hold'); between disconnected keyframes (connectedRight:false) → just left. getSequencedValues() then deserializeAndSanitizes left/right (falling back to the prop's default) and calls propConfig.interpolate(left, right, progression) — e.g. linear lerp for numbers, OKLab-space blending for rgba, left-biased for strings/booleans — writing the result into a values atom keyed by prop path.

7. Playback. sheet.sequence lazily builds a Sequence whose length and subUnitsPerUnit (fps grid) are prisms derived from project historic state (sanitized: default length 10s, default 30 fps). The sequence holds a swappable IPlaybackController in an Atom. sequence.play(conf) validates range/rate/iterationCount/direction, then DefaultPlaybackController.play() schedules ticks on a dataverse Ticker (from the core rafDriver or a user rafDriver): each tick computes elapsedRealTime * rate, maps it into the range honoring direction and iteration count, and writes the new position into its state Atom. Because interpolationTripleAtPosition reads sequence.positionPrism, advancing the playhead reactively re-derives every sequenced prop, which propagates up through getValues() to your onChange listeners. sequence.position = t pauses and jumps. attachAudio() calls replacePlaybackController to swap in an AudioPlaybackController so position is driven by the Web Audio clock for tight A/V sync.

8. Tickers / rafDrivers. coreTicker.ts lazily creates the default rafDriver (requestAnimationFrame loop in browsers, single setTimeout in SSR) and exposes getCoreTicker(). A dataverse Ticker becomes active (calling the driver's start) only when something is subscribed, and dormant (stop) when idle. createRafDriver() lets users own the tick source for cross-library sync, XR, or deterministic stepping.

The public/private pattern

Every public class (TheatreProject, TheatreSheet, TheatreSheetObject, TheatreSequence, and the rafDriver public api) is a façade that delegates to an internal class via privateAPI()/setPrivateAPI() (a WeakMap in privateAPIs.ts). The internal classes (Project, Sheet, SheetObject, Sequence) hold all the dataverse state and logic. Studio receives the privateAPI resolver as part of CoreBits (from CoreBundle.getBitsForStudio) so it can reach into internals to read tracks, write keyframes, and manage state — without core ever depending on studio.

Gotchas & footguns (12)
  • Importing @theatre/core has side effects: it registers a global on window/global immediately. Importing it twice (different copies in node_modules, or a bundler that fails to dedupe) throws 'module @theatre/core is loaded more than once'. The package sets sideEffects:true in package.json for this reason.
  • Core and studio are decoupled by design — core never imports studio. They communicate only through the window.__TheatreJS_CoreBundle / __TheatreJS_StudioBundle globals. The studio type inside core is just a placeholder ($____FixmeStudio), so a lot of studio-touching code is loosely typed.
  • In production (core-only) you MUST pass config.state to getProject, or nothing animates. Without state and without studio, the browser throws after ~1s and SSR logs an error. The state's definitionVersion must equal globals.currentProjectStateDefinitionVersion ('0.4.0') or validation throws.
  • getProject(id, config) is memoized by id. Calling it twice with a *different* config throws in dev. Same applies to sheet.object(key, config): re-declaring with different props throws unless you pass {reconfigure:true}.
  • Each sheet has exactly ONE sequence named 'default'. There is no multi-sequence-per-sheet API; sheet.sequence is it. Sequence length defaults to 10s and fps (subUnitsPerUnit) to 30 if the stored values are missing/invalid.
  • Composite props (compound, enum) are not directly sequenceable — only their simple leaf descendants get keyframe tracks. getArrayOfValidSequenceTracks filters out tracks whose prop path no longer maps to a sequenceable prop (e.g. after a config change), silently dropping stale tracks.
  • sequence.play() before project.ready resolves is a no-op that warns (and resolves true) — when studio is loaded the project state loads asynchronously, so wrap play() in project.ready.then(...). In pure core mode the project is ready almost immediately.
  • SheetObject.getValues() recomputes on EVERY change to ANY (even nested) prop, including every frame during playback. It is kept fast by ordering merge layers cheapest-first and caching each deep-merge in a WeakMap. initialValue is intentionally NOT layered last despite being mutable, because it's rarely used (mostly @theatre/r3f) — a known tradeoff (linear P-208).
  • deserializeAndSanitize() returning undefined is meaningful: it signals 'stored value invalid for this prop type' and the system falls back to the prop's default. It's used both for static overrides and for keyframe left/right values, not just at config time.
  • rgba interpolation happens in OKLab color space (not naive RGB lerp), so two-keyframe color animations won't match a simple channel-wise interpolation. Strings/booleans/file/image default to 'leftInterpolate' (hold the left value) unless you supply a custom interpolate().
  • Keyframes are stored as a PointableSet (byId/allIds) in state, but interpolation needs them sorted by position; getSortedKeyframesCached memoizes the sort+clone. Don't assume state stores them in order.
  • Pointers are typeless at runtime — obj.props.position.x is just {root, path:['position','x']}. val()/onChange() throw if you pass something that isn't a pointer (or, for onChange, a prism). The reactivity comes entirely from dataverse prisms, not from the pointer objects themselves.

The dev-only visual editor UI for Theatre.js (AGPL); attaches to core at runtime and owns project state, with undo/redo, scrubbing, panels, and an extension API.

TL;DR

@theatre/studio is the visual editor you use during development to keyframe and tweak animations. Sits one layer above @theatre/core as an OPTIONAL, dev-time peer. Core never imports studio; the two find each other at runtime through window globals. Core concepts

StudioBundle ↔ CoreBundle handshake
Studio and core are independent modules that find each other via window globals. Studio publishes window.__TheatreJS_StudioBundle; core publishes window.__TheatreJS_CoreBundle. Whichever loads second calls registerCoreBundle, which exchanges CoreBits (the projects pointer, coreExports, and the privateAPI resolver). No user wiring is needed and core never imports studio — that is exactly why production bundles can drop the AGPL studio.
Three state layers: historic / ahistoric / ephemeral
Studio state is split three ways. HISTORIC = the undoable document (project defs, keyframes, static overrides, selection) backed by saaz, persisted. AHISTORIC = UI config (panel positions, visibility, pins) persisted to IndexedDB but never in undo history. EPHEMERAL = runtime-only (loaded extensions, pane instances, props being scrubbed), lost on reload. Each is a dataverse Atom exposed via pointers (studio.atomP.historic, .ahistoric, studio.ephemeralAtom).
Transactions & scrubbing
studio.transaction(fn) applies an undoable mutation immediately. studio.scrub() returns a Scrub whose capture()/recapture() can run many times during a drag, then commit() (undoable) or discard() (rollback). Both funnel through transactionApi.set(pointer, value), which writes a keyframe (if the prop has a sequence track) or a static override otherwise.
Saaz-backed store & undo/redo
StudioStore delegates to a saaz SaazFront: edits go through saaz.tx()/tempTx(), saaz maintains the optimistic local state and the undo/redo stack, and saaz.subscribe pushes the new state.cell into the studio Atom that powers the UI. undo()/redo() delegate to saaz; the history survives reloads.
Shadow DOM isolation
The editor mounts a React root inside an open ShadowRoot on a fixed-position container (react-shadow), so the page's CSS can't leak into the studio UI and vice-versa. styled-components targets the shadow root; theme tokens are CSS variables on :host. In visual-regression tests a plain div replaces the shadow root.
Panels: Detail & Sequence editor
The Detail panel inspects the selected SheetObject, rendering one editor per prop (simple editors for primitives, compound editors for objects). The Sequence editor is the timeline: a DopeSheet shows keyframe dots per track, a GraphEditor shows interpolation curves; dragging the playhead scrubs, and keyframe edits go through stateEditors on the sequence.
Extensions & panes
studio.extend(extension) contributes toolbar buttons, toolsets, and pane classes. PaneManager.createPane(className) instantiates a pane (descriptor stored in historic state) and the UI renders it reactively. @theatre/r3f's snapshot editor is the canonical extension.
Selection
studio.setSelection(items) writes the outline selection into historic state; Detail and Sequence panels react to it. onSelectionChange() lets extensions (e.g. r3f) observe which sheet objects are selected.

Deep dive — how it works internally

Startup & attach

  1. Importing @theatre/studio runs index.ts, which creates the singleton Studio and publishes a StudioBundle on window.__TheatreJS_StudioBundle.
  2. @theatre/core publishes window.__TheatreJS_CoreBundle. Whichever loads second calls studioBundle.registerCoreBundle(coreBundle), which calls coreBundle.getBitsForStudio() to receive CoreBits (the projects pointer, coreExports, and the privateAPI resolver).
  3. The user calls studio.initialize() (or theatre.init({studio:true})), then studio.ui.render() mounts the editor.
  4. For every project (now and future), Project.attachToStudio() runs: it reconciles on-disk vs. browser-persisted state, then repoints the project's PointerProxies at studio.atomP.historic.coreByProject[projectId] — making studio the live owner of state.

State model

Studio state lives in StudioStore, split into three layers, each a dataverse Atom:

  • historic — the saaz-backed, undoable document (keyframes, static overrides, project defs, selection). Persisted.
  • ahistoric — UI config (panel geometry, visibility, pins). Persisted to IndexedDB, never undoable.
  • ephemeral — runtime-only (extensions, pane instances, props being scrubbed). Lost on reload.

UI components subscribe to pointers into these atoms via useVal/usePrism (from @theatre/react), so they re-render only when their slice changes.

Editing: transactions and scrubs

studio.transaction(({set, stateEditors}) => { set(obj.props.x, 100) }) commits an undoable change immediately. studio.scrub() is for interactive drags: capture(({set}) => set(obj.props.position, v)) can be called repeatedly while the mouse moves, then commit() (or discard()). Internally both go through StudioStore.transaction/tempTransaction → saaz.tx/tempTx. The transactionApi.set resolver (createTransactionPrivateApi.ts) checks whether the prop has a valid sequence track: if so it writes/updates a keyframe (setKeyframeAtPosition); otherwise it writes a static override (setValueOfPrimitiveProp). On commit, saaz advances its state and undo stack, saaz.subscribe fires, and StudioStore._atom.set(state.cell) updates the studio Atom — which is the very thing core's prisms read, so the change re-flows to the screen.

UI rendering

studio.ui.render() creates a fixed-position container, attaches an open Shadow DOM, and mounts a React root rendering UIRoot: the global toolbar, PanelsRoot (panes from PaneManager), command overlay, and toast notifier, all wrapped in ProvideStyles (styled-components scoped to the shadow root). The Detail panel and Sequence editor read selection + project state from the studio Atom and write back through transactions/scrubs.

Persistence & cloud sync

Historic state persists through saaz's storage adapter; ahistoric state and auth tokens persist to IndexedDB (IDBStorage, persistAtom). Auth.ts runs an OAuth device flow, stores tokens in a BroadcastChannel-synced atom (cross-tab), and the SyncStore links (AppLink/SyncServerLink, tRPC) connect to the cloud backend. Cloud sync is optional — studio works fully offline with everything in IndexedDB.

Undo/redo

All historic mutations go through saaz's transaction system, which keeps an undo/redo stack. studio.undo()/redo() delegate to StudioStore → saaz; history is persisted and survives reloads.

  • Import studio right after core. The CoreBundle/StudioBundle handshake relies on both being on window; if studio is imported late, they may not connect.
  • Studio is the OWNER of state once attached, not core. Core reads it through PointerProxies that attachToStudio() repoints at studio.atomP — which is why edits reach the screen with no extra wiring.
  • studio.transaction() is undoable by default; scrubs are NOT permanent until you commit(). A discarded scrub never enters undo history.
  • transactionApi.set chooses keyframe-vs-static by whether a valid sequence track exists at that path — not by what the editor looks like. A compound pointer recurses into every simple sub-prop and requires all of them.

  • The UI lives in an open Shadow DOM. It's isolated from page CSS, but apps can still inject into it — beware global style collisions if you mount portals inside.

@theatre/react + theatric

React bindings that subscribe components to dataverse prisms/pointers, plus a zero-boilerplate Leva-style "useControls" entry point.

TL;DR

@theatre/react (packages/react) is a tiny set of React hooks (useVal, usePrism, usePrismInstance, useAtom) that bridge the dataverse FRP library to React, re-rendering a component only when the specific prism/pointer/atom value it reads actually changes. theatric (packages/theatric) builds on top of @theatre/core + @theatre/studio to offer a single hook, useControls(), that spins up a hidden Theatre project/sheet/object and gives you a tweakable control panel with zero setup, persistence, and undo/redo. The two packages are layered: theatric uses dataverse pointers/val internally but does its own React state plumbing rather than reusing @theatre/react's hooks.

Where it fits

@theatre/react sits at the boundary between the dataverse reactivity engine (packages/dataverse: Atom, prism, pointer, val) and React's render cycle. It depends only on @theatre/dataverse (plus lodash-es and queue-microtask) and has react/react-dom as peer deps; it is consumed by @theatre/r3f, the studio UI, and any app reading Theatre object props reactively. theatric sits one layer higher as a developer-facing convenience wrapper: it depends on @theatre/core, @theatre/studio, and @theatre/react (workspace deps), and targets quick prototyping rather than full timeline animation. Nothing inside Theatre depends on theatric; it is a leaf/entry package.

Core concepts

Pointer
A typed, lazy path object (e.g. atom.pointer.foo.bar or obj.props.x) describing WHERE a value lives, not the value itself. Pointers are proxy objects you keep narrowing; you read them with val(pointer). theatric's $get/$set take a selector function p => p.some.path that returns a pointer.
Prism
dataverse's cached, composable reactive computation (FRP). prism(fn) memoizes fn's result and tracks which pointers/prisms it read. getValue() freshens and returns the current value; onStale(cb) fires cb when a dependency may have changed (without recomputing). React hooks subscribe via onStale and recompute via getValue.
Atom
dataverse's mutable state container. Exposes .pointer (root pointer), .prism, set()/setByPointer(). useAtom wraps one so the component can mutate state without re-rendering, and selectively read sub-paths reactively via useVal(atom.pointer.x).
Fine-grained re-render via stale+freshen queue
Instead of re-rendering on every dependency change, _usePrismStaticInstance subscribes with onStale, enqueues itself, and during a batched microtask calls getValue(); it only forceUpdates if the new value !== lastValue. So a component re-renders only when its specific computed value actually changes.
Order-sorted global update queue
Every prism instance gets an incrementing order (parents mount first => smaller order). The shared queue is kept sorted by order so refreshes run top-down (ancestors before descendants), which avoids React's 'zombie child' / stale-props problem and over-rendering. The queue is flushed in a single queueMicrotask wrapped in unstable_batchedUpdates.
theatric useControls()
A single React hook that declaratively describes controls ({name:'Andrew', age: types.number(28)}); under the hood it reconfigures one Theatre object per 'panel', subscribes to object.onValuesChange, mirrors values into React useState, and returns the live values plus imperative $get/$set helpers and folder namespacing.
Dev-only studio / treeshaking
theatric imports studio but guards all studio usage behind process.env.NODE_ENV checks (init({studio:true}) and maybeTransaction) so the studio UI and write transactions are stripped from production bundles, leaving only read-only Theatre core.

# @theatre/react — subscribing React to dataverse

The package is small (one file, ~440 lines) but the mechanism is subtle. There are four public hooks and one internal engine.

The hooks (thin wrappers)

  • useVal(pointerOrPrism) is literally usePrism(() => val(p), [p]). val() (packages/dataverse/src/val.ts) resolves a pointer to a prism via pointerToPrism, a prism via getValue(), or returns a plain value. So useVal(obj.props.foo) reactively reads a single Theatre/Atom sub-path.
  • usePrismInstance(der) is usePrism(() => der.getValue(), [der]) — for when you already hold a prism.
  • usePrism(fn, deps, debugLabel) is the real workhorse (see below).
  • useAtom(initialState) just useRefs a new Atom(initialState) so it is stable across renders. Crucially, holding the atom does NOT subscribe the component — you only re-render where you call useVal(atom.pointer.x). This is the 'state without re-render' pattern in the docstring example.

usePrism internals

usePrism keeps a useRef holding three things: an inner Atom that stores the latest fn, a stable prism that reads atom.prism.getValue()() (i.e. calls whatever fn is current), and the last deps. On each render it diff-checks deps (length, then element-by-element with !==); if deps changed it calls atom.set(fn). Because the wrapper prism reads the fn through the atom's prism, swapping the fn marks the wrapper stale automatically — that is how a closure capturing fresh props/deps gets re-evaluated. The stable prism instance is then handed to _usePrismStaticInstance.

The subscription engine: _usePrismStaticInstance + the queue

This is where fine-grained rendering happens. On first render it builds a QueueItem stored in a ref:

  • assigns a monotonically increasing order (lastOrder++) — since parents render/mount before children, ancestors get smaller orders;
  • calls der.onStale(cb) immediately (turning the prism HOT before render, a deliberate perf change noted in the long comment to avoid slow 'cold reads'). The cb calls queueUpdate() which pushToQueue(item).
  • pushToQueue inserts into a module-global queue array kept SORTED by order (binary-ish findIndex for the first item with order >= item.order), backed by a Set for dedupe, then schedules one queueMicrotask.

When the microtask fires, runQueue() runs inside unstable_batchedUpdates so all the resulting React updates are batched into one commit. It shifts items in order (ancestors first) and for each calls item.der.getValue(). The key move: it compares newValue !== item.lastValue. It only calls runUpdate() (which is the forceUpdate) when the value actually changed. This is the 'freshen prisms before render' design: a dependency going stale does NOT guarantee a re-render — only an actual value change does (the docstring's isPositiveD example where setting 1→2 keeps it positive and must NOT re-render).

If getValue() throws (a possible 'zombie child' reading stale props), it logs and defers that one item to a normal React render by calling runUpdate() directly, where reading happens safely during render.

During each render, _usePrismStaticInstance also (a) removes itself from the queue if it was queued but is rendering anyway (avoids double work), (b) reads der.getValue() fresh and stores it as lastValue, and (c) in a useLayoutEffect cleanup marks unmounted = true, calls untap() (the onStale unsubscribe) and removes itself from the queue. runUpdate checks unmounted so a queued update can't force-update a gone component.

Net effect: each component subscribes only to the exact prism/pointer it reads, re-renders only on real value changes, and the global order-sorted queue guarantees top-down refresh in a single batched microtask — matching React concurrent-mode expectations without over-rendering.

# theatric — zero-boilerplate controls

theatric does NOT reuse @theatre/react's hooks; it does its own React/Theatre plumbing in packages/theatric/src/index.ts.

Bootstrapping

At module load, in development only, it getStudio().then(() => init({studio: true})) to mount the studio UI, and defines maybeTransaction as either a real getStudioSync(true)!.transaction (dev) or a no-op (prod). These process.env.NODE_ENV === 'development' static guards let bundlers treeshake studio out of production. A lazily-created singleton project is built by callGetProject() → getProject('Theatric', _projectConfig). initialize(config) just stores the config before the project is first created (warns if already initialized) and resolves on project.ready.

useControls(config, options)

  1. Memoizes config against a ref using isEqualWith(..., equalityCheckWithFunctionsAlwaysEqual) so that re-creating the config inline each render (and functions like button onClicks) does not thrash.
  2. Splits config into controlsWithoutButtons (real props) and buttons (entries with type:'button', created by the exported button(onClick) helper). Optional folder namespaces the props under {[folder]: props} and prefixes button/action keys.
  3. Maintains module-global registries allProps[panel] and allActions[panel] (arrays), because MULTIPLE useControls calls can target the same panel and must be merged into one Theatre object. It calls sheet.object(panel, Object.assign({}, ...allProps[panel], props), {reconfigure: true, __actions__...}) — reconfigure: true lets the same object be redefined with merged props. This happens both during render (so first values aren't undefined) and inside a useEffect that pushes/splices this instance's props/actions on mount/unmount and reconfigures accordingly.
  4. Bridges Theatre → React: seeds useState from object.value (unwrapping folder), then in a useEffect subscribes object.onValuesChange((newValues) => …). Because reconfigure can fire the callback redundantly, it guards with isEqual(newValues, valuesRef.current) to prevent infinite loops, then setValues.
  5. Returns {...values, $set, $get}. $get/$set (memoized) take a selector p => pointer, resolve it against the object's props (or the folder sub-pointer), validate with isPointer, and either val(pointer) (get) or wrap api.set(pointer, value) in maybeTransaction (set, giving undo/redo via studio in dev).
  6. selectObjectIfNecessary auto-selects the first created object in the studio so new users see their controls immediately.

getAssetUrl(asset) proxies project.getAssetUrl (throws if not initialized/ready), and types is re-exported straight from @theatre/core.

Gotchas & footguns (12)
  • usePrism (and therefore useVal) requires a deps array just like useMemo. If you read props/state inside fn but omit them from deps, the prism reads stale values; the runQueue error handler even mentions a missing dep as a likely cause of thrown getValue() errors.
  • Holding an Atom via useAtom does NOT cause re-renders. You must read a specific sub-path with useVal(atom.pointer.x) to subscribe. Mutate via atom.set / atom.setByPointer.
  • A dependency going 'stale' is not enough to re-render — _usePrismStaticInstance only forceUpdates when der.getValue() !== lastValue. Pure recomputations that yield the same value are intentionally swallowed (the isPositiveD example).
  • The update queue, lastOrder, and microtask flag are MODULE-GLOBAL singletons shared by every component in the bundle. Ordering correctness depends on parents mounting before children so they get smaller order values.
  • Updates are flushed in a queueMicrotask wrapped in react-dom's unstable_batchedUpdates — so changes don't render synchronously; they coalesce into one batched commit on the next microtask.
  • react-dom is a peer dependency of @theatre/react specifically because it imports unstable_batchedUpdates from 'react-dom'. The @types pin to React 19 (this checkout is in the middle of a React 19 migration per git status).
  • theatric does NOT use @theatre/react's hooks; it manages its own useState/useEffect bridge to object.onValuesChange. Don't assume the two share the subscription engine.
  • theatric relies on process.env.NODE_ENV === 'development' static checks to treeshake studio. Studio UI, init({studio:true}), and $set transactions only exist in development; in production $set's maybeTransaction is a no-op and changes won't persist via studio.
  • object.onValuesChange can fire twice with identical values when the object is reconfigured (which useControls does constantly to merge panels). Both the value bridge (isEqual guard) and config memoization exist purely to break the resulting infinite-loop / churn.
  • All theatric controls across the whole app land in ONE Theatre object per panel ('Default panel' by default) inside a single hidden 'Theatric' project/'Panels' sheet. Use the folder option (or panel) to avoid key collisions when the same component renders multiple times.
  • $get/$set take a SELECTOR function (p => p.path) that must return a pointer, not a plain value path — passing a non-function or a non-pointer throws explicitly.
  • initialize() must be called before the first useControls() (which lazily creates the project). Calling it afterward warns and is ignored, since _projectConfig is only read at project creation time.

@theatre/r3f

React-three-fiber bindings that make THREE.js objects animatable in Theatre.js and add a visual snapshot editor to Studio.

TL;DR

@theatre/r3f is the glue between react-three-fiber and Theatre.js. Its core is the editable() factory: you wrap an r3f element (e.g. editable.mesh / e('mesh','mesh')) and it auto-creates a Theatre "sheet object" with animatable props (position, rotation, scale, color, light/camera params) that drive the underlying THREE.Object3D. It also ships a separate Studio extension (@theatre/r3f/dist/extension) that registers a "snapshot editor" pane: a second r3f canvas showing a clone of your scene with selectable proxies, TransformControls gizmos, helpers, a grid, and a reference window for direct manipulation. The package is split into a main/ runtime (shipped to apps) and an extension/ editor (loaded only inside Studio), which talk to each other through a shared zustand store via deliberately-named private exports.

Where it fits

Sits at the top of the Theatre.js stack as an optional integration package. Depends on @theatre/core (sheets, sheet objects, types, studio API, rafDriver), @theatre/react (useVal), @theatre/dataverse (prism/val/onChange for the toolbar), react-three-fiber/three/three-stdlib, and zustand. Nothing in the monorepo depends on it; it's consumed by end-user r3f apps. The main entry is bundled into the user's app bundle; the extension entry is a peer that's only studio.extend()-ed into the Studio editor. The two halves are intentionally in different bundles and communicate through ____private_* / __private_* exports from the main entry (a shared singleton zustand editorStore).

Core concepts

editable() factory & e
editable (often imported as e) is both a function and a namespace. editable.mesh (or e('mesh','mesh'), or e(MyComponent,'mesh'), or e.primitive) returns a forwardRef React component. It renders the underlying r3f element but, as a side effect, registers a Theatre sheet object whose props animate that element. The two args are (Component, type): the second type selects which entry of the factory config (and thus which props/icon/helper) to use. For drei/custom components you pass the component as the first arg and a type string as the second.
EditableFactoryConfig / ObjectConfig / PropConfig
A static registry (defaultEditableFactoryConfig) maps each supported r3f element to an ObjectConfig: a props map plus metadata (icon, useTransformControls, dimensionless, optional updateObject, optional createHelper). Each prop is a PropConfig with three responsibilities: parse(props) converts the incoming JSX prop into a plain value (handling arrays, scalars, Vector3, scale-x style keys), apply(value, object) writes that value onto the THREE object (e.g. object.position.set(...)), and type is the Theatre shorthand prop definition (number/rgba/compound) that defines the animatable knob and its default/nudge multiplier.
Sheet object lifecycle (r3f -> Theatre)
In editable.tsx's effects: on mount it builds a storeKey from the sheet address + theatreKey, calls sheet.object(theatreKey, {...config props...}) to create the ISheetObject, adds it to allRegisteredObjects and the editorStore's editables map. It then records sheetObject.initialValue from the parsed JSX props, and subscribes via sheetObject.onValuesChange(setFromTheatre, rafDriver) so that whenever Theatre's value changes it re-applies every prop to the object and calls invalidate() to request an r3f re-render. On unmount it detaches the object and cleans up the store.
editorStore (shared zustand singleton)
A single vanilla zustand store in main/store.ts is the bridge between the app runtime and the Studio extension. The runtime writes into it (live scene/gl via bindToCanvas, the editables map via each editable). The extension reads from it through useExtensionStore (which wraps the same store instance imported as the private ____private_editorStore). It also holds sceneSnapshot (a scene.clone()) and editablesSnapshot that the editor renders, so the editor shows a frozen copy, not the live scene.
Snapshot / proxy editor
The editor never manipulates the real scene. createSnapshot() clones the live scene; ProxyManager clones that again and traverses it, turning each object tagged userData.__editable into an EditableProxy portaled into its original parent. Selecting a proxy sets the Studio selection to its sheet object; dragging the TransformControls gizmo opens a studio.scrub() and writes position/rotation/scale back into the sheet object's props, which (because the same sheet object drives the live object) animates the real scene. refreshSnapshot/<RefreshSnapshot/> re-clones so async-loaded models appear.
Studio extension / pane / toolset
The extension is a plain IExtension object registered with studio.extend(extension) from @theatre/r3f/dist/extension. It contributes: a global toolbar icon that calls studio.createPane('snapshot'); a snapshot pane class whose mount({node}) boots a React root rendering SnapshotEditor; and toolsets (global, snapshot-editor) built reactively with dataverse prism/onChange that read/write the editor sheet object inside studio.transaction. The toolset is rendered into the pane via studio.ui.renderToolset('snapshot-editor', container).
dimensionless objects & helpers
Lights, cameras and fog have no visible geometry, so their ObjectConfig is marked dimensionless and given a createHelper (CameraHelper, *LightHelper, BoxHelper). In the editor these get an overlay icon (react-icons) and an invisible click-target Sphere so they remain selectable, and the helper is added to the scene while hovered/selected. updateObject (cameras call updateProjectionMatrix()) runs after props are applied.

Deep dive — how it works internally

Two halves, one store

The package is physically split into main/ (runtime, bundled into the user's app via the . export) and extension/ (the Studio editor, the ./dist/extension export). They never import each other's internals directly except through three deliberately-named private exports from index.ts: ____private_editorStore, __private_allRegisteredObjects, and __private_makeStoreKey. The contract is the singleton zustand editorStore in main/store.ts — both bundles end up referencing the same module instance at runtime, so the runtime can push state in and the editor can read it out.

The editable() factory (main/editable.tsx)

createEditable(defaultEditableFactoryConfig) returns editable. Object.assign(editable, extensions) attaches one ready-made component per config key (editable.mesh, editable.group, editable.spotLight, …) plus editable.primitive. Calling editable(Component, type) directly lets you wrap arbitrary components (this is how the drei cameras are made, and how the playground's e('points','mesh') works).

Each editable component is a forwardRef that, on render:

  1. Reads the current sheet from useCurrentSheet() (thrown error if no <SheetProvider> above) and an optional rafDriver from useCurrentRafDriver().
  2. Computes storeKey = makeStoreKey({...sheet.address, objectKey: theatreKey}).
  3. In a useLayoutEffect, calls sheet.object(theatreKey, props) where props is built by spreading every config[type].props[*].type (the Theatre shorthand prop definitions) plus any user additionalProps. The created ISheetObject is added to allRegisteredObjects (a WeakSet used later to recognize "our" objects in Studio selection) and registered in editorStore.editables[storeKey] together with its objectConfig, type, and a visibleOnlyInEditor flag. If the object already exists it re-creates with {reconfigure: true} (so changing additionalProps updates the schema).
  4. Another effect computes sheetObject.initialValue by running every prop's parse(props) over the JSX props — this seeds Theatre with the values you wrote in JSX.
  5. The main subscription effect calls setFromTheatre(sheetObject.value) once and then sheetObject.onValuesChange(setFromTheatre, rafDriver). setFromTheatre iterates the config props and calls value.apply(newValues[key], object) to mutate the THREE object, runs the optional updateObject(object), and calls invalidate() so r3f (which the editor and apps run in frameloop="demand") re-renders. Cleanup detaches the object, removes it from the WeakSet, and removes it from the store.
  6. It renders <Component ref={mergeRefs([objectRef, ref])} {...props} userData={{__editable:true, __storeKey:storeKey}} />. The userData tags are how the editor later finds editable objects when traversing a cloned scene. Note visible is special-cased: visible="editor" makes the object visible only inside the editor.

Prop configs (editableFactoryConfigUtils.ts + defaultEditableFactoryConfig.ts)

baseObjectConfig gives every transformable object position/rotation/scale (vector props) with a BoxHelper and useTransformControls:true. extendObjectProps layers on extra props for lights (intensity/distance/decay/color), cameras (near/far, plus fov/zoom on perspective), spotlights (angle/penumbra), fog, etc., each with the right helper and icon. createVectorPropConfig is notable: its parse handles array, scalar (broadcast to xyz), Vector3 objects, and r3f's scale-x shorthand keys, falling back to a default with a notify.warning. Color props convert THREE Color <-> Theatre rgba.

SheetProvider + binding (main/SheetProvider.tsx, store.ts)

<SheetProvider sheet={...}> puts the sheet in context AND, via useLayoutEffect, calls bindToCanvas({gl, scene}) (pulled from useThree). bindToCanvas -> editorStore.init(scene, gl) stores the live scene/gl and immediately calls createSnapshot() so that if the editor pane is already open it refreshes.

The Studio extension (extension/index.ts)

Registered by the app via studio.extend(require('@theatre/r3f/dist/extension').default). It declares:

  • A global toolbar button ("Create Snapshot") that calls studio.createPane('snapshot').
  • A snapshot-editor toolset: transform mode (translate/rotate/scale), transform space (world/local), shading (wireframe/flat/solid/rendered), and a refresh-snapshot button. These are built with dataverse prism reading the editor sheet object and pushed through onChange; clicking them runs studio.transaction(({set}) => set(editorObject.props.transformControls.mode, value)).
  • A snapshot pane class whose mount({node}) does ReactDOM.createRoot(node).render(<SnapshotEditor paneId/>) and unmounts on the next tick.

editorStuff.ts lazily creates a dedicated R3F UI project sheet with an Editor sheet object holding all the viewport/transform UI state — so the editor's own settings are themselves Theatre state.

The snapshot editor render path (SnapshotEditor -> ProxyManager -> EditableProxy)

SnapshotEditor renders a shadow-DOM-isolated (react-shadow + styled-components) second <Canvas frameloop="demand">. Inside, EditorScene adds an InfiniteGridHelper (a custom GLSL3 shader-material plane), an axesHelper, the editor camera (useSnapshotEditorCamera, which is a drei PerspectiveCamera + a patched OrbitControls bound to a Editor Camera <paneId> sheet object so the editor viewpoint is itself animatable/scrubbable with debounced commits), the shared helpersRoot group, and <ProxyManager>.

ProxyManager takes sceneSnapshot from the store, clones it, and traverses it. For every object with userData.__editable that still exists in editables (dedupes duplicate keys), it creates an EditableProxy portaled (createPortal) into the object's original parent, wrapped in a PortalWrapper to avoid duplicate React keys. It also captures each mesh's original material so it can swap materials for the shading modes. When an object is selected (via useSelected, which watches studio.onSelectionChange and matches against allRegisteredObjects) and its config has useTransformControls, it mounts a <TransformControls> gizmo; dragging it opens a studio.scrub() on pointer-down and capture(({set}) => set(sheetObject.props.position, ...)) on every change, committing on pointer-up. Because that's the same sheet object the live editable subscribes to, manipulating the gizmo animates the real scene.

EditableProxy handles selection clicks (studio.setSelection([sheetObject])), hover/selection helpers (added to the scene), and overlay icons + an invisible click-sphere for dimensionless objects. It also subscribes the proxy object itself to the sheet object so the proxy mirrors animated values.

Reference window

ReferenceWindow copies the user's real canvas (gl.domElement) into a small 2D canvas every frame via drawImage, giving a live picture-in-picture of the actual rendered output next to the editor viewport. It requires gl={{preserveDrawingBuffer:true}} on the user's Canvas, and shows a warning if not set.

Cloud / sync — @theatre/saaz · sync-server · app

Theatre.js's newer local-first, optimistic state-sync engine (saaz) plus the Next.js account backend (app) and the WebSocket realtime relay (sync-server).

TL;DR

@theatre/saaz is a generic local-first state-management + sync engine split into a SaazFront (in the editor/browser) and a SaazBack (authoritative server). The front applies edits optimistically to local state and keeps a queue of unacknowledged "transactions", pushing them to the back over a transport (in production, tRPC-over-WebSocket); the back replays them deterministically, advances a clock, and broadcasts the new snapshot so all peers reconcile. @theatre/sync-server is a standalone WebSocket server that hosts one SaazBack per project and exposes it via tRPC; @theatre/app is a Next.js backend handling user accounts, teams/workspaces, and OAuth device-flow auth for the studio. Studio's StudioStore wraps a SaazFront so the entire editor state becomes a synced document.

Where it fits

These are the "1.0 cloud" pieces of the monorepo, layered on top of the existing editor. @theatre/studio is the primary consumer: StudioStore (packages/studio/src/StudioStore/StudioStore.ts) instantiates a SaazFront, feeding its state.cell into a dataverse Atom so the whole studio UI reactively follows synced state, and studio.transaction() becomes a saaz tx. saaz depends only on @theatre/dataverse (atoms/prisms), immer, idb, and @theatre/utils; it is transport-agnostic via the SaazBackInterface. sync-server depends on saaz (SaazBack), tRPC, ws, prisma, and calls into app (appClient) for auth. app depends on Next.js, next-auth, prisma, and tRPC; it owns the user/team/workspace data model and mints the JWT access tokens that sync-server verifies. The studio reaches app for auth (device flow) and reaches sync-server for state sync.

Core concepts

SaazFront vs SaazBack (the front/back split)
saaz is one engine in two halves. SaazFront runs wherever edits originate (the browser/studio). SaazBack is the single source of truth. They speak through the SaazBackInterface. When co-located (tests, SSR) the front holds a direct SaazBack reference; in production the front holds a tRPC proxy that forwards each interface method over WebSocket to a SaazBack living inside sync-server.
Optimistic updates + reconciliation
A tx() is applied locally immediately (optimism) and appended to optimisticUpdatesQueue tagged with a per-peer peerClock. The front's optimisticState prism = backend snapshot + replayed queued updates. The back replays each update, bumps its global clock and the peer's lastIncorporatedPeerClock, and returns the new snapshot. On the next _setSessionState, the front garbage-collects any queued update whose peerClock <= lastIncorporatedPeerClock — that's the reconciliation handshake.
Deterministic transactions
Updates store *recorded editor invocations* (name + args) and draftOps, not the resulting state, so the same transaction can be replayed on front and back and yield identical results. _createTransaction even re-runs the editor twice in dev and throws if results differ. Non-determinism (random/ids) is tamed by GeneratorSpy, which records generator outputs into the transaction for replay.
op snapshot vs cell (the two state layers)
A FullSnapshot has {op, cell}. op is a plain immer-managed object mutated by named editors. cell is the rogue.ts branch/box structure mutated by a freeform draftFn(draft) and diffed into Ops. Studio puts its entire state into cell (via cellShape: StudioState) and leaves op/editors essentially empty — so studio edits flow through the draft/Ops path, and Ops carry forward+backward ops enabling undo/redo.
Clocks, peers, and sessions
The back has one monotonic clock. Each client is a peerId (per-tab, nanoid) with its own peerClock. SessionState records backendClock, lastIncorporatedPeerClock, and the last snapshot. Multiple tabs = multiple peers under one user account; presence (tempTransactions) is broadcast so peers see each other's in-progress edits (withPeers prism).
Closed/crashed session recovery
Unacknowledged updates are persisted per-peerId to FrontStorage. On startup, _acquireLocksOnClosedSessions uses the Web Locks API to take over peerIds that no live tab holds, loads their pending updates as closedSessions, and flushes them to the backend before they're GC'd. This is how offline/crashed edits eventually sync.
Two backends, two prisma DBs, JWT trust
@theatre/app (Next.js, postgres) owns accounts/teams/workspaces and issues RS256 JWTs via the OAuth device flow. @theatre/sync-server (ws server, its own postgres with just ProjectState) hosts the saaz backs. sync-server doesn't share app's DB; it verifies the studio's access token against app's published public key (/api/jwt-public-key) and can call appClient.studioAuth.canIEditProject for authorization.

Deep dive — how it works internally

The shape of saaz

saaz is a generic engine; nothing in packages/saaz knows about Theatre. The schema (Schema<OpSnapshot, Editors, Generators, CellShape>) plugs in the domain: editors are named reducer functions over the immer op draft, generators are deterministic-ized helpers, and cellShape types the freeform cell. The studio schema lives in packages/sync-server/src/state/schema.ts (stateEditors, initialState, and the exported schema) and is imported by both sync-server and studio.

The data layers: op and cell

Every state is a FullSnapshot<Op> = {op, cell}.

  • op is mutated by editors via immer in applyOptimisticUpdateToState (shared/transactions.ts). Editors are invoked by *path string* (get(schema.editors, fnPath.split('.'))).
  • cell is the rogue.ts structure. rogue wraps a base object in an immer draft behind a Proxy; you mutate it like a normal object, then compare() walks before/after and emits Ops (ChangeType, SetBoxedValue). Each value is a Cell with a $type (map/boxed/deleted) and $branches. Branch names get regenerated when a prop switches type so concurrent edits to the same key don't collide. compare is run in both directions to produce forwardOps and backwardOps — the latter powering undo. fromOps replays Ops onto a base. jsonFromCell/current convert a cell back to plain JSON.

A transaction's life (SaazFront.tx)

  1. _createTransaction records editor calls (recordInvokations — a proxy that captures [path, opts] without executing), and runs draftFn against a makeDraft(cell) to get draftOps + backwardOps.
  2. In dev it replays the transaction twice and diffs, throwing on non-determinism (naming the culprit editor). GeneratorSpy ensures rand()/id-generators return recorded values on replay.
  3. _pushOptimisticUpdate bumps peerClock, appends a Transaction to optimisticUpdatesQueue, and (if undoable) pushes {forwardOps, backwardOps} to the undo stack.
  4. Reactive prisms recompute: base (backend snapshot) → optimisticState (+ closed-session updates + queued updates, skipping any already incorporated) → withTemps (+ live temp transactions) → withPeers (+ other peers' presence temps). state returns finalState(withPeers). Studio's subscribe callback copies state.cell into its Atom.

Push/pull/reconcile

  • _reflectUpdatesToBackend (a subscribeDebounced over a prism) computes the updates not yet incorporated and calls backend.applyUpdates({backendClock, peerId, updates}). It drains closed sessions first, then the current session.
  • SaazBack.applyUpdates (back/SaazBack.ts) replays each update via applyOptimisticUpdateToState(..., playbackOnly=true) onto _dbState, advances _clock and the peer's lastIncorporatedPeerClock, then calls subscribers with shouldCheckForUpdates:true.
  • The front's _processBackendUpdate/_setSessionState install the new snapshot and GC acknowledged updates from the queue.
  • _subscribeToBackend pulls on connect and whenever the back signals an update; presence flows the same channel.
  • Persistence: _reflectBackendStateToStorage and _reflectOptimisticUpdatesToStorage mirror state into FrontStorage; allSyncedToFrontStorage/waitForStorageSync gate on it.

How sync-server hosts the back

packages/sync-server/src/index.ts runs an http+ws server with tRPC's applyWSSHandler serving syncServerRouter. projectStateRouter maps each SaazBackInterface method to a tRPC procedure; saaz_subscribe is a tRPC observable bridging saaz's callback subscription to a WS stream. getSaazBack(dbName) (saaz/index.ts) lazily creates one SaazBack per project, in-memory (BackMemoryAdapter). Every procedure first calls ctx.requireValidSession, which verifyAccessTokenOrThrow (utils/authUtils.ts) — fetching app's RS256 public key and verifying the studio's JWT. sync-server has its own prisma client (ProjectState{id,data}) and an appClient tRPC proxy back into app for richer authz (canIEditProject).

How app fits in

@theatre/app is a Next.js app-router project. Two tRPC surfaces: the normal appRouter (server/api/root.ts: projects/workspaces/teams/me, next-auth-guarded via protectedProcedure) consumed by app's own UI, and the studioTrpcRouter (server/studio-api/root.ts) used by the studio — notably studioAuth (OAuth2 device flow: deviceCode→tokens→refreshAccessToken, plus canIEditProject) and syncServerUrl which currently returns a hardcoded ws://localhost:3001/api/trpc. Auth is GitHub OAuth via next-auth + a prisma adapter; the device flow lets studio (running on any origin) obtain a JWT whose public key sync-server can verify. Prisma/postgres holds users, teams, workspaces, guest access, and DeviceAuthorizationFlow.

How studio consumes all this (relation to SyncStore)

packages/studio/src/StudioStore/StudioStore.ts constructs a SaazFront with the studio schema. On the server/SSR it uses a direct SaazBack; in the browser it uses createTrpcBackend(dbNamePromise), which implements SaazBackInterface by forwarding each method to getStudio().authedLinks.syncServer.projectState.saaz_*. That syncServer link is a SyncServerLink (packages/studio/src/SyncStore/SyncServerLink.ts) — a tRPC WS client whose URL comes from app's syncServerUrl. AppLink/SyncServerLink under studio/src/SyncStore/ plus Studio.ts's authedLinks are the glue: AppLink talks to app (auth + config), SyncServerLink talks to sync-server (state). So the chain is: studio UI → StudioStore.transaction → SaazFront.tx → optimistic local state (Atom) + queue → createTrpcBackend → SyncServerLink (WS, JWT) → sync-server projectStateRouter → getSaazBack → SaazBack → broadcast → all peers' SaazFront.

Gotchas & footguns (9)
  • These are pre-release/in-flux (@theatre/saaz 0.7.0; app & sync-server 0.1.0). Lots of TODOs, dead/commented code, and $IntentionalAny/$FixMe casts. ensureStateIsUptodate is a stubbed no-op (migrations commented out); BackStorage/BackMemoryAdapter don't actually persist (the back is purely in-memory and resets on restart).
  • getUpdatesSinceClock always returns the *entire* snapshot (type:'Snapshot'), never a diff (the Diff branch is 'todo'). This is O(state size) per sync and a known scaling limitation.
  • Transactions MUST be deterministic. Editors are replayed multiple times; in dev SaazFront throws if a transaction invokes different editors/args or produces different output across runs. Any randomness/id-generation must go through schema.generators so GeneratorSpy can record/replay it — calling Math.random() directly inside an editor will break collaborative consistency.
  • In StudioStore the storage adapter is currently forced to FrontMemoryAdapter (the IDB branch is unreachable: ... || true), and dbName is hardcoded to 'test'. So persistence/offline is wired but effectively disabled in the current studio integration — do not assume edits survive a reload yet.
  • Studio uses the cell (draft) path, not op/editors. IStateEditors is {} and schema.editors is {}; all studio mutation happens via draftFn → rogue Ops. Undo/redo therefore depend on rogue's backwardOps, not on inverse editors.
  • Two separate prisma clients / databases / generated-client paths: app (packages/app/prisma/client-generated, full account model) and sync-server (packages/sync-server/prisma/client-generated, just ProjectState). They do NOT share a DB — trust flows via JWT verification of app's public key, plus tRPC calls from sync-server to app.
  • syncServerUrl is hardcoded to ws://localhost:3001/api/trpc in studio-api/root.ts, and canIEditProject currently looks up a Workspace by id with the user check commented out (// userId). Authorization is stubbed/permissive in this snapshot — not production-safe yet.
  • peerId is per-tab and security-sensitive: the comment in SaazFront notes the backend is supposed to bind peerIds to accounts so they can't be stolen and GC stale ones, but that enforcement isn't fully present in SaazBack here.
  • The Web Locks API (navigator.locks) is used to take over closed/crashed sessions' pending updates. This only works in secure browser contexts and is skipped in non-browser environments; closed-session recovery silently does nothing server-side.

  • globs src/{shared,personal,tests}/*/index.html with fast-glob to build the rollup input map, plus a top-level src/index.html whose index.tsx renders a PlaygroundPage listing every playground (grouped via import.meta.glob);
  • crucially sets resolve.alias from getAliasesFromTsConfigForRollup(), which turns @theatre/core → <repo>/packages/core/src/index.ts, etc. This means Vite treats the monorepo packages as source, not externals — so editing any package's src instantly reflects in the running playground with no separate build. optimizeDeps.exclude keeps core/studio out of pre-bundling in dev for the same reason.
  • injects definedGlobals (from core/devEnv) and a process.env.BACKEND_URL resolved from packages/app/.env (defaults to https://app.theatrejs.com if absent).

To hack: create packages/playground/src/personal/<name>/index.html + index.tsx (personal/ is gitignored) or use an existing one under src/shared/ (camera, dom, three-basic, r3f-rocket, theatric, notifications, etc.).

How building works

  • yarn cli build (→ tsx devEnv/cli.ts build) first runs yarn run build:ts (the aggregate tsc --build devEnv/typecheck-all-projects/tsconfig.all.json, emitting declarations) then, sequentially, yarn workspace <pkg> run build for the publishable set: @theatre/dataverse, @theatre/saaz, @theatre/core, @theatre/studio, @theatre/react, @theatre/r3f, theatric, @theatre/browser-bundles.
  • Each package's own build typically = run-s build:ts build:js build:api-json: build:ts = tsc --build (types/.d.ts + declarationMap), build:js = tsx ./devEnv/build.ts (esbuild bundling), build:api-json = api-extractor run (rolls .d.ts + emits API JSON for docs). See packages/utils/package.json for the canonical shape.
  • browser-bundles is the <script>-tag build: packages/browser-bundles/devEnv/build.ts runs esbuild twice — an IIFE dist/core-and-studio.js (src/core-and-studio.ts imports core + studio and assigns window.Theatre = core) and a minified IIFE dist/core-only.min.js (src/core-only.ts). Consumers then do Theatre.core / Theatre.studio.

How tests run

  • Unit/integration (Jest): yarn test from root. jest.config.js matches packages/*/src/**/*.test.ts and devEnv/**/*.test.ts, transforms via jest-esbuild, maps @theatre/* aliases via getAliasesFromTsConfigForJest, stubs css/svg/png and a few ESM-only deps, and runs a studio integration-test setup file. yarn test --watch for watch mode. (A separate jest.compat-tests.config.js + yarn test:compat:* exercises the published-package compatibility fixtures.)
  • End-to-end (Playwright): live in packages/playground/src/tests/<name>/ as *.e2e.ts next to their playground's index.html/index.tsx. yarn test:e2e → yarn workspace playground run test → playwright test --config=devEnv/playwright.config.ts. That config's webServer boots the playground (yarn run serve:ci on port 8082, reusing an already-running instance) before the suite. yarn test:e2e:ci runs the dot-reporter chromium-only CI variant. Some .e2e.ts files also do visual-regression screenshots (expect(page).toHaveScreenshot()), which are only stable in the repo's linux Docker container (docker-compose.yml).

Typecheck & lint

  • yarn typecheck (alias of yarn build:ts) = tsc --build devEnv/typecheck-all-projects/tsconfig.all.json, a project-references graph covering every package. --watch supported.
  • yarn lint:all = eslint . --ext ts,tsx --ignore-path=.gitignore --rulesdir ./devEnv/eslint/rules. The --rulesdir loads the custom no-relative-imports rule. .eslintrc.js is intentionally minimal (unused-imports, tsdoc/syntax, consistent type-imports, switch-exhaustiveness, no-floating-promises, ban raw lodash in favor of lodash-es). --fix autofixes (including rewriting ../ imports to aliases).
  • Pre-commit: husky's .husky/pre-commit runs yarn lint-staged (eslint --fix with the custom rulesdir on changed ts/tsx, prettier --write on all) plus a dataverse precommit docs-regen check.

benchmarks & dataverse-experiments

  • benchmarks is a dev-only harness run with yarn benchmarks (→ tsx devEnv/serveBenchmarks.ts, esbuild dev server on port 8087). It can A/B current source against @theatre/*@0.5.0 pulled from npm to spot perf regressions.
  • dataverse-experiments is an experimental, source-only fork/sandbox of dataverse (Atom, Box, Ticker, derivations/prism). It has no build/JS step — only typecheck (tsc --build) — and is not published; it's a scratchpad for reactive-dataflow ideas.
Gotchas & footguns (11)
  • CONTRIBUTING.md is out of date on prerequisites: it says node 14+ and yarn 1.22, but root package.json requires node>=18 and pins yarn 3.6.3. Trust package.json.
  • npm install is unsupported — this is a Yarn workspaces repo. Use yarn (which auto-delegates to the bundled yarn 3 via .yarnrc.yml/packageManager).
  • Don't forget yarn postinstall after yarn — it sets up husky hooks and runs prisma generate for the cloud app/sync-server. The root postinstall script references @theatre/app and @theatre/sync-server, so a fresh install touches the cloud packages even if you only care about the animation library.
  • The playground README references a live-reload esbuild server at ./devEnv/build.ts, but the playground has since migrated to Vite (packages/playground/vite.config.ts, port 8082). The build.ts described in the README no longer exists; trust vite.config.ts.
  • The dev loop works without building packages ONLY because Vite/Jest/ESLint alias @theatre/* to each package's src/index.ts via tsconfig.base.json paths. If you add a new package or alias, update tsconfig.base.json paths AND add a project reference in devEnv/typecheck-all-projects/tsconfig.all.json, or typecheck/aliasing will silently miss it.
  • Two different test systems with confusingly similar names: .test.ts = Jest (root yarn test), .e2e.ts = Playwright (root yarn test:e2e, lives inside the playground). They're configured and run completely separately.
  • The custom ESLint rule requires the --rulesdir ./devEnv/eslint/rules flag; running plain eslint without it won't load no-relative-imports. The root scripts and lint-staged already pass it.
  • Default Playwright e2e runs chromium-only with a tight 4s per-test timeout; firefox is commented out in playwright.config.ts. Visual-regression screenshots only match in the repo's linux Docker container, not on Mac/Windows.
  • @theatre/utils is "private": true (never published) and imports are written as @theatre/utils/... subpaths (e.g. @theatre/utils/types), not relative paths — enforced by the no-relative-imports rule.
  • yarn cli release enforces a clean git tree, runs typecheck+lint (unless --skip-lint), assigns one shared version across all main packages (r3f is the exception with its own version), and tags + npm-publishes. Don't run it casually; @theatre/r3f releases on its own schedule.
  • yarn passes extra args through to the underlying script, so yarn test --watch, yarn typecheck --watch, and yarn test:e2e --debug / --project=firefox all work.

Glossary

The Theatre.js domain vocabulary in one place.

Project
Top-level container, obtained via getProject(id, {state}) and deduped per id. Holds historic state (keyframes/overrides) and ephemeral state in dataverse Atoms wrapped in PointerProxies. attachToStudio() swaps those proxies onto studio's Atom.
Sheet
An instance of a SheetTemplate under a Project (project.sheet(name, instanceId='default')). Owns its SheetObjects and exactly one Sequence. Can be instantiated multiple times sharing one template.
Sheet Object
sheet.object(key, propsConfig) — a typed animatable object whose .props is a reactive Pointer tree matching the prop config. Internal SheetObject computes the final value via the four-layer merge in getValues(). Multiple sheet instances share one SheetObjectTemplate.
Sequence
The single timeline/playhead per sheet (sheet.sequence). Holds a swappable IPlaybackController in an Atom; exposes positionPrism (the reactive playhead) that feeds keyframe interpolation. Length defaults to 10s, fps (subUnitsPerUnit) to 30.
Prop / PropTypeConfig
An animatable property, declared with t.number/string/boolean/rgba/compound/etc. Each PropTypeConfig carries a default, deserializeAndSanitize() (validates stored JSON), and interpolate(). compound() turns shorthand ({x:0}) into longhand. Only simple leaf props are sequenceable; compound/enum are not.
Atom
dataverse's mutable holder of immutable state, with per-path change tracking via a lazily-grown Scope tree. Writes (set/setByPointer) use structural sharing (updateDeep) so identity-based (===) diffing notifies only the exact subscribed paths.
Pointer
A type-safe, referentially-stable reference to a path inside an Atom (e.g. obj.props.position.x), carrying only {root, path} metadata — not a value. Implemented as a JS Proxy with WeakMap caching so p.a.b === p.a.b. You resolve it with val()/pointerToPrism().
Prism
dataverse's lazy, cached, push-based derivation built from a function that auto-discovers its dependencies as it reads pointers/prisms. Cold (no subscribers, recomputes each read) vs hot (cached, maintains a dependency graph); fresh vs stale. Composes and recomputes only along real dependency edges.
val()
The universal read primitive. Normalizes a pointer, prism, or plain value into a value. Inside a running prism the read is TRACKED as a dependency; outside one it's a one-off non-reactive snapshot.
Ticker
dataverse's scheduler that decouples 'changed' from 'run subscriber'. onChange schedules a refresh onto the ticker; tick(t) drains all callbacks once per frame, collapsing duplicate work (batching). Usually rAF-driven; goes dormant after ~180 empty ticks.
rafDriver
A tick source that feeds a Ticker (default backed by requestAnimationFrame). createRafDriver() lets you supply your own (sync with r3f/gsap/XR or manual deterministic stepping). onChange() and sequence.play() accept one to control update rate.
PointerProxy
A swappable pointer indirection wrapping a pointer in an Atom so its target can change. Project's historic/ephemeral pointers are PointerProxies; attachToStudio() setPointer()s them onto studio's Atom, and reading prisms re-track to the new source automatically.
Studio
The optional visual editor (@theatre/studio). Attaches to core at runtime via the window-global handshake, becomes the owner of project state, and edits values through scrub/transaction APIs. Dropped in production builds.
Scrub
A studio temp transaction that can be re-captured many times (e.g. while dragging a value), then committed (undoable) or discarded. Both scrub.capture and transaction funnel through transactionApi.set, writing a static override or a keyframe.
Transaction
In studio: an immediately-applied, undoable mutation of the studio state. In saaz: a recorded set of editor invocations + draftOps tagged with a peerClock, replayed deterministically on front and back for collaborative consistency.
Pane
A dockable UI panel in the studio. Extensions register pane classes whose mount({node}) renders React into a DOM node (e.g. @theatre/r3f's 'snapshot' pane renders the SnapshotEditor).
Extension
An IExtension object registered with studio.extend(). Contributes toolbar buttons, toolsets, and pane classes. @theatre/r3f/dist/extension is the canonical example, adding the 3D snapshot editor.
editable() (r3f)
The @theatre/r3f factory (often imported as e). editable.mesh / e(Component,'type') wraps an r3f element, auto-creating a Theatre sheet object whose animatable props drive the underlying THREE.Object3D every frame. Must live under a <SheetProvider>.
useControls() (theatric)
theatric's single hook for a zero-boilerplate, Leva-style control panel. Declares controls inline; under the hood reconfigures one hidden Theatre object per panel and bridges value changes into React useState, with $get/$set, folders, buttons, persistence, and undo/redo.
saaz / SaazFront / SaazBack
The 1.0 local-first sync engine. SaazFront (browser) applies edits optimistically and queues unacknowledged transactions; SaazBack (server, in sync-server) is the authoritative replica that deterministically replays them, advances a monotonic clock, and broadcasts snapshots for reconciliation.
CoreBundle / StudioBundle handshake
The runtime mechanism by which core and studio find each other WITHOUT importing each other. Core publishes window.__TheatreJS_CoreBundle; studio publishes window.__TheatreJS_StudioBundle; the second to load calls registerCoreBundle, exchanging CoreBits (projectsP, coreExports, privateAPI).
privateAPI / public facade
The two-class pattern throughout core: every user-visible class (TheatreProject/Sheet/SheetObject/Sequence) is a thin facade delegating to an internal class via a WeakMap (privateAPI/setPrivateAPI). Keeps the public surface stable while internals evolve; studio reaches internals through privateAPI in CoreBits.
Four-layer value merge
How SheetObject.getValues() computes a prop's final value: deep-merge of defaults → initialValue → static overrides → sequenced keyframes, ordered cheapest-to-most-volatile and WeakMap-cached so only the changed (usually sequenced) layer recomputes per frame.
interpolationTripleAtPosition
Core's keyframe interpolation: given a track pointer and the sequence's positionPrism, returns a prism of {left, right, progression} for the current playhead by selecting the bracketing keyframes and solving the bezier/hold easing. This is what makes a keyframe override a static value during playback.
